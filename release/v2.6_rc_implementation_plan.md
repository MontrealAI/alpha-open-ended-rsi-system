# Nova-Seeds v2.6.0-rc.1 Release Candidate Plan (Implementation-Ordered)

## Section 1: Current repo assessment

### 1.1 Verified current state (from repository)
- The repo already carries **v2.6.0-rc.1** posture in executable surfaces:
  - FastAPI title/version: `backend/app/main.py` sets `"Nova-Seeds v2.6 RC API"` and `"2.6.0-rc.1"`.
  - SDK package version is `2.6.0-rc.1` in `sdk/package.json`.
  - Release checklist exists at `release/v2.6/checklist.md`.
- The constitutional sequence is present in docs and release framing, and should stay explicit as:
  **identity → proof → settlement → governance**.
- Core architecture is intact and should remain so:
  - Solidity contracts in `contracts/`
  - FastAPI + Postgres backend/indexer in `backend/`
  - SDK bindings/helpers in `sdk/`
  - Operator dashboard in `dashboard/`

### 1.2 What is already implemented and useful for RC hardening
- **Release/provenance baseline exists**:
  - CI workflow: `.github/workflows/ci.yml`
  - Release provenance workflow: `.github/workflows/release-provenance.yml`
  - Deterministic manifest generator: `scripts/release/generate_provenance_manifest.py`
  - Verification guide: `docs/verify-release.md`
- **Backend governance/proof surfaces exist**:
  - `/ready`, `/proof/summary`, governance endpoints, and `/metrics` in `backend/app/main.py`.
  - Reorg-aware indexing and cursor semantics in `backend/app/indexer.py`.
  - additive migration path in `backend/migrations/001_init.sql` and `backend/migrations/002_v26_hardening.sql`.
- **Schema baseline exists** for v2.6 threshold/decryption payloads in `schemas/v2.6/` with examples.
- **Dashboard baseline exists** in `dashboard/` with summary/proof/governance tabs and export UX.

### 1.3 Missing pieces that block a stronger v2.6 RC gate
1. **Checklist is not machine-enforced**
   - `release/v2.6/checklist.md` exists, but CI does not currently fail when checklist sections drift or are incomplete.
2. **Provenance manifest file coverage is broad but not explicit for release-control docs**
   - `scripts/release/generate_provenance_manifest.py` includes `docs/**/*.md`, but does not explicitly call out `release/v2.6/checklist.md` and `RELEASES.md` as required control artifacts.
3. **No seed-scoped proof detail API for operator triage**
   - Backend has `/proof/summary` only; no deterministic per-seed detail endpoint in `backend/app/main.py`.
4. **No schema-enforced governance evidence packet**
   - `docs/proof-docket-template/` is template-driven, but there is no canonical JSON schema for a governance adjudication packet.
5. **Dashboard has no automated smoke checks**
   - No `dashboard/tests/` script currently validating API contract compatibility.
6. **No single operator runbook that sequences migration + backfill + verification + rollback**
   - guidance is spread across `docs/verify-release.md`, backend README, and checklist.

## Section 4: Acceptance criteria for v2.6 tag

A v2.6.0-rc.1 tag is acceptable only when all are true:

1. **No widened claim surface**
   - docs and release notes keep RC posture (not audited/final/trustless claims).
2. **Constitutional sequence preserved**
   - identity → proof → settlement → governance remains explicit in touched docs.
3. **Checklist is machine-enforced in CI**
   - validator runs and fails on missing required sections.
4. **Provenance bundle is reproducible and complete**
   - source archive, SBOM, SHA256SUMS, provenance manifest, attestation, and release-control docs are packaged.
5. **Governance evidence packet is schema-versioned**
   - schema + canonical example validate in tests.
6. **Backend exposes deterministic seed-level proof detail**
   - endpoint exists, tested for present and absent seed data.
7. **Indexer reorg/idempotency protections remain intact**
   - existing regression tests continue to pass.
8. **Dashboard exposes proof state clearly**
   - complete/incomplete/unknown states render without JS failure.
9. **Unified operator runbook exists and is linked**
   - migration/backfill/verification/rollback flow is in one operator-facing doc.

## Section 5: Risks / unknowns / assumptions

### Risks
- **Event-shape risk**: available event payloads may not support all desired seed-level proof diagnostics without additional contract events.
- **Operational drift risk**: CI workflow commands and runbook commands may diverge unless both are updated together.
- **UI contract risk**: dashboard could silently break if backend payload shape changes without smoke tests.
- **Migration safety risk**: rollback guidance is additive-focused but not fully exercised end-to-end in tests.

### Unknowns
- Whether RC validation must cover one chain deployment or multiple test networks.
- Whether governance evidence packet signatures require single operator, multisig, or council quorum in this phase.
- Whether seed-level proof status should remain informational in v2.6 RC or include strict timing/SLO guarantees.

### Assumptions
- v2.6 remains a **proof and governance hardening release candidate**.
- Existing stack and architecture are preserved (Solidity + FastAPI/Postgres + SDK + dashboard).
- Work is delivered as small implementation-ordered PRs, with release/provenance gating first.
